# Review: 2026-07-23-code-templates.md

Review of `/Users/adityabhargava/agency-lang/worktree-code-templates/docs/superpowers/plans/2026-07-23-code-templates.md`.

Overall: the plan is well structured, honest about its own gaps (the self-review notes are good), and the line references I spot-checked are accurate (the `#` description parser really is at `lib/parsers/parsers.ts:1224-1242` with wiring at `:1270` and `:1359`; AG8 really is free in `lib/typeChecker/diagnostics.ts:591-598`; the double-`return interrupt` idiom in Task 8 matches `writeAST` at `stdlib/agency.agency:452-457`, which I initially thought was a typo — it isn't). But there are four problems that will stop an executor cold, and several more that will produce a feature that doesn't do what the plan says it does. Blocking items first.

---

## Blocking: the plan has no way to make an expression-sized `Code` value

Tasks 11 and 13 build their tests on fragments like `code("tmp")`, `_parseAST("42")`, and `_parseAST("\"hello\"")`. These do not parse. I verified: `pnpm run ast` on a file containing just `42` fails with "Failed to parse Agency program: expected ... an assignment, a function call, ..." — the top-level grammar accepts statements and declarations but not bare expressions. (`const inner = 5` at top level *does* parse, so Task 12's statement fragments are fine.)

This is not just a test-authoring bug. It is a hole in the design of `fill`:

- `isCode` recognizes only `{ type: "agencyProgram" }`, and `nodesFor` grafts `program.nodes` (plan lines 1173-1179, 1238-1241). A program's `nodes` are top-level items. There is no representation for "a `Code` value that is one expression," so an `expr` hole can never legally be filled with `Code` — the thing Task 13's "accepts a Code filler of the right type" test requires.
- There is also no Agency-level way to *produce* a fragment: `loadTemplate` reads whole files, and the existing `parseAST` (`stdlib/agency.agency:426`) parses whole programs.

The plan needs an explicit fragment story before Task 9: an expression-parsing entry point (the expression parser exists; it needs exporting behind something like `_parseExpr`), and a `Code` shape that records whether it holds a program, a statement list, or one expression, so `nodesFor` can check fragment kind against hole sort. Right now "Code values are grafted as trees" is specified only for the one sort (`statements`/`decl`) where a program's nodes happen to fit.

## Blocking: the hygiene design cannot pass its own tests

Two independent defects in Task 11, both visible by reading the plan's sketches against the plan's tests:

1. **Collision detection compares the wrong sets.** `bindersOf` collects *binders* (assignments). The flagship test fills `#userExpr` with `code("tmp")` — a *use* of `tmp`, which binds nothing — so `fillerBinders` is empty, no collision is detected, and the "renames the template binder when a filler uses the same name" test fails even after Step 4 is complete. Capture avoidance needs template binders intersected with filler **free names**, not filler binders. (Set aside that `code("tmp")` doesn't parse — see the previous item; fix that first and this failure becomes visible.)

2. **Renames are applied to the template only.** Step 4 computes `renamed = renameBinders(ast, renames)` and then substitutes the *original, unrenamed* filler values into it. So for the filler-vs-filler case — `setup` and `cleanup` both declaring `tmp` — the collision is detected (both are binders, so detection works here), a rename is computed, applied to the template (which never mentions `tmp`, a no-op), and both fillers are grafted still declaring `tmp`. The "resolves collisions between two fillers" test fails. Fillers need their own rename maps, applied to each grafted tree, with distinct fresh names per filler.

The direction of protection also matters and only one direction is handled even in intent: the design renames so a filler's *use* can't capture a template binder. The reverse — template code after a hole using a name the filler happens to declare — is exactly Task 12's "bindings are local to the hole" rule, and nothing implements it (next item).

## Blocking: Task 12's "bindings are local to the hole" rule is stated but never implemented

Task 12 declares: "A `statements` hole is a block, so whatever it declares is invisible after it." No step makes that true. `substitute` splices grafted statements inline into the surrounding statement list (plan lines 1196-1211), so in the completed program a filler's `const inner` shares the enclosing scope and *is* visible to everything after the hole. Agency has no anonymous block construct to wrap them in (blocks exist only as bodies of `if`/`while`/`def`/etc.), so this can't be a one-line fix either. The plan's own test doesn't catch it — it only asserts the output contains `const inner = 5`, which is true whether or not scoping holds.

Decide what the rule actually is: (a) a *template-checking* rule — the checker treats code after a hole as unable to reference names the hole might introduce, with runtime leakage accepted; or (b) a real isolation guarantee — which needs hygiene to rename **all** filler binders unconditionally (not just colliding ones), contradicting Task 11's "renaming is selective." Either answer is defensible; the plan currently promises (b) and implements neither.

Separately, the Task 12 execution test is a no-op twice over:

- `holeReturnTemplate.agency` has an explicit `return value`, so it never trips the definite-return check regardless of the exemption. The plan's own Step 3 text admits this ("its only return is fine").
- The check ships at `"warn"` by default (`docs/dev/typechecker/definite-returns-remaining-work.md:23`, `lib/typeChecker/definiteReturns.ts`), so even a template that trips it wouldn't fail a run. The predicted Step 3 failure won't happen, and Step 4's fix is unobservable at default config.

Replace the fixture with a template whose only `return` sits inside a `#statements` hole, and state the config under which the check is an error. Also note the exemption code imports `lib/runtime/template/holes.js` into `lib/typeChecker/` — check whether the typechecker imports from `lib/runtime/` anywhere today; `findHoles` is a pure AST walk and may belong somewhere both layers can reach.

## Blocking: the execution-test format cannot express "expect compile error AG8001"

Tasks 7 and 13 write `.test.json` files with `"expectedError": "AG8001"`. The runner has no such key: `TestCase` in `lib/cli/test.ts:36-39` carries `expectedOutput` and evaluation criteria, and nothing in `lib/cli/test.ts` handles an expected-error field — a compile failure surfaces as a test *execution* error, not a pass. The plan hedges by pointing at `tests/agency/validation/validateAnnotationFail.test.json` as "the nearest example ... the key name there is authoritative," but that file contains no error key at all — it's a program that catches its own failure and returns `"failed-as-expected"`, a pattern unavailable here because the failing thing is compilation of the test file itself.

Options: extend the runner with an expected-compile-error mode (new work — add it as a task), or drop these to vitest tests that call the compile path directly and assert on the diagnostic. Either is fine; the plan just has to pick one, because as written Task 7 Step 6 can never show "PASS."

## Important: `Code` is never defined, and the codebase already owns most of it

The plan's Agency signatures use a `Code` type (`holesOf(template: Code)`, `fill(template: Code, ...)`) but no task introduces it. Meanwhile `stdlib/agency.agency:420-424` already defines `type AST = { type: "agencyProgram", nodes: any[], docComment?: any }`, and `parseAST` / `writeAST` / `format` already ship "an AST is an ordinary first-class value you can hold, pass, and print." Task 8's architectural framing ("resist any temptation to make it a special form") is arguing for something the codebase already did.

So the altitude question: is `Code` just `AST` under a new name? If yes, say so and reuse it — `loadTemplate` is then read-file-plus-`parseAST` plus the sandbox interrupt, and the genuinely new surface is holes, `fill`, hygiene, and checking. If `Code` is meant to be distinct (e.g. to carry the fragment-kind field from the first blocking item, or to be opaque so users can't hand-build one), that's a real design decision the spec should own, with a task that creates the type. Related consequence either way: since `isCode` accepts any `{ type: "agencyProgram" }` object, `fill(t, { v: parseAST(modelOutput) })` is a two-call injection path — deliberate, user-opt-in, but the Task 16 guide should say it out loud.

## Important: `fill(template: Code, values: Json)` can't accept the grafting case

`Json` is primitives/arrays/records; a `Code` value is none of those (or worse, structurally sneaks in as a record and defeats the point of the signature). The Agency-level signature contradicts the feature's second bullet ("`Code` values are grafted as trees"). The values parameter needs something like `Record<string, Json | Code>`.

## Important: the `expectedType` field does not survive printing, contradicting the plan's own claim

Task 13 Step 3 justifies storing `expectedType` on the node because it "keeps the type travelling with the node through printing and re-parsing, which Task 15 [sic — 14] depends on." False as specified: `formatHole` prints sigil, name, and `typeAnnotation` only. A `Code` value serialized as source (Task 14) and re-parsed loses every `expectedType`, so a template checkpointed mid-workflow comes back unable to type-check its fills. Either the printed form must carry it (a grammar change — probably wrong), or accept the loss and recompute expected types by re-running the template check on load/deserialize. Pick explicitly.

## Important: Task 14's predicted failure probably doesn't exist — verify before building

A `Code`/`AST` value is a plain JSON-serializable object (no class instances, no functions), and `parseAST` results can already be held across interrupts today. *Unverified inference:* I did not run a checkpoint test, but I see no mechanism in the plan's own description by which plain-object state would fail to round-trip — the state stack's registered-serializer machinery exists for values that JSON can't carry (function refs, class instances). Run Task 14 Step 1 first, as the plan structure already suggests, and if it passes, the task collapses to keeping the test. Don't build the serializer to satisfy a failure prediction that never reproduces.

## Important: Task 1 misses a documentation user of `#` descriptions

`docs/site/guide/llm.md:52-53` uses the syntax being removed (`capital: string # the capital city of the country`). After Task 1 the guide documents a parse error. Migrate it alongside the two test files, and grep `docs/` for stragglers as part of Step 8's verification rather than trusting this list.

## Smaller items

- **Task 16 names a directory that doesn't exist.** The linter lives at `lib/linter/`, not `lib/lint/`. `lib/lsp/` is correct.
- **Task 15's driver won't run as sketched.** `run(source, "main")` — `run` takes a `CompiledProgram` (`stdlib/agency.agency:108-110`); the right call is `runCode(source)` (`:308`). `writeAgency` doesn't exist (the plan hedges correctly), and whatever replaces it, its output goes straight to `runCode` — the `format(...)` wrapper is redundant. Also `runCode` returns a `Result`, so the driver must unwrap before returning or the exact-match expectation fails.
- **Splices in expression position are unspecified.** After Task 6, `const x = #...items` parses (splice is consumed before sort is assigned), and at fill time the spread only fires when the hole sits in an array — which call-argument lists are, so `f(#...args)` half-works by accident. State which sorts allow `splice` and reject it elsewhere at parse or fill time.
- **Identifier holes in import specifiers need a shape check.** Task 10 replaces the hole with a `{ type: "variableName" }` node inside the import-specifier list. Confirm import specifiers actually hold nodes of that shape (they may be plain strings or `{ name, alias }` records) and that the generator prints them; also note the lint work found imports historically lacked `loc`, which the round-trip tests may trip over.
- **`renameBinders` is scope-blind.** It rewrites every `variableName` matching a colliding name anywhere in the template — including inside unrelated functions that legitimately use their own `tmp`. Not wrong (the renamed program is α-equivalent within each scope only if the name was bound at top level of that scope... it isn't in general). Worth a test: two functions each binding `tmp`, one hole in one of them.
- **Task 5's decl-hole test may be order-sensitive.** `firstHole` walks object keys recursively; if the identifier-hole wiring in `def #name` runs through the same `holeParser`, the import-specifier test's hole and the top-level `#helpers` hole both depend on which alternation wins first. Fine if it works, but if these tests flake, the walk order is why.

## Anti-pattern audit (against docs/dev/anti-patterns.md)

Checked every code sketch in the plan against the catalog. The parser sketches are clean (and the catalog explicitly exempts parsers from the ordering rule), but the runtime sketches hit four entries.

**Duplicating existing code — the catalog's first entry, and the plan's worst offense.** The catalog's own bad example is a hand-rolled recursive AST walker, and the plan writes *four* of them: `visit` in `holes.ts` (Task 8), `substitute` in `fill.ts` (Task 9), and `collect` plus `allNames` in `hygiene.ts` (Task 11), plus a fifth in the Task 4 test helper. Meanwhile `walkNodesArray` exists at `lib/utils/node.ts:570`, and `lib/stdlib/agency.ts:262` already carries a comment calling it "the single source" for AST walking — `_walkAST` at `:270` is built on it. `findHoles`, `bindersOf`, and `allNames` should each be a `walkNodesArray` filter, not a bespoke recursion. (One nuance to preserve: `findHoles` deliberately doesn't descend into a hole; check whether `walkNodesArray` supports pruning, and if not, that one walker may earn its existence — say so in a comment.) Same entry, second hit: `isLegalIdentifier` in `agencyGenerator.ts` (Task 6 Step 5) and `LEGAL_IDENTIFIER` in `fill.ts` (Task 10 Step 3) are the same regex defined twice, each with its own "mirrors varNameChar" comment. Define it once, next to `varNameChar` in the parser module, and import it from both places — three copies of a rule that must stay in sync via comments is how it drifts.

**Imperative code everywhere — yes, the plan has this, concentrated in one place.** To answer the direct question: the plan's *module structure* is genuinely declarative — `holesOf`, `liftValue`, `fillHoles`, `bindersOf`, `renameBinders` are small named operations with clear contracts, and callers use them declaratively. The failure is Task 11 Step 4, which pastes ~35 lines of raw mechanism inline at the top of `fillHoles`: three mutable accumulator arrays built by push loops, manual dedup via `!colliding.includes(binder)`, duplicate detection via `indexOf !== lastIndexOf`, and a mutable `counter` threaded through a rename-building loop. That is the "how" sitting undigested inside the one function whose job is to express the "what" (validate → detect collisions → rename → substitute). The whole block should be one declarative call — `const renames = computeRenames(ast, values)` — living in `hygiene.ts` with the other hygiene mechanism, built from expressions like `templateBinders.filter(name => fillerFreeNames.includes(name))` rather than accumulate-and-push. This also matters for the review's blocking hygiene finding: the fix (free names, per-filler renames) means rewriting this block anyway, so specify it as a named function with a contract instead of inline wiring. Smaller instances of the same pattern: `holeNames`'s manual first-occurrence dedup loop (Task 8) and the dedup push inside `collect` (Task 11) — both are a filter-on-first-index one-liner as in the catalog's own "good" example.

**Order-dependent mutable state.** Same Task 11 Step 4 block: `colliding` must be populated by both loops before the rename loop runs, and `counter` couples rename names to loop iteration order. Deriving `renames` as a single expression from the collision list (`colliding.map((name, i) => ...)`) removes the ordering hazard. The parser tasks' ordering (statement-hole before expression-statement, Task 4) is fine — that's the documented parser exemption, and the plan correctly calls out that order *is* the tie-break rule there.

**Leaky abstractions.** Two hits. First, every literal node in `liftValue` (Task 9) is hand-assembled with `as unknown as AgencyNode` double-casts — six of them — encoding the parser's private node shapes in a second location. The plan even admits the shapes are unverified sketches (self-review note 3). If literal-node constructors exist anywhere in `lib/`, use them; if not, building them (`makeStringLiteral(value, loc)` etc., co-located with the node types) turns six unchecked casts into one typed seam and gives the shapes a single owner. Second, Task 13's `expectedType` field is checker-internal state stored on a public AST node that then silently evaporates at the print boundary — internal detail leaking into the interface, with the breakage already covered in the `expectedType` finding above.

**Checked and clean:** no nested ternaries (the `formatHole` ternaries are single-level), no silent catch blocks, no one-line ifs, no magic numbers, no dynamic requires, and — worth stating because this feature invites it — the catastrophic-test-failure entry does not apply: the Task 9 injection test only generates and string-matches, never executes, and if Task 15's lifting broke, the generated program would call an unimported `readFile` in a `runCode` subprocess and fail with an unknown-function error rather than actually reading anything.

## Test-plan audit: will these tests fail when the code breaks?

Three questions per test: does it test what it claims, does it fail when the code breaks, and what's not tested at all. The strong points first: the Task 9 injection test (positive containment plus a negative regex), the two-round-trip stability tests, and Task 15's exact-match expectation are all genuinely load-bearing — they fail loudly on the failures they target. The problems fall into three groups.

### Tests that pass while the code is broken

- **The number/boolean/array lift tests (Task 9) can't tell a value from a string.** `toContain("42")` passes whether the filler was lifted to the number literal `42` or wrongly to the *string* literal `"42"` — the printed string contains `42` either way. Same for `toContain("true")` and `toContain("[1, 2]")`. A lifter that stringifies everything passes all three. Fix: assert the unquoted form and the absence of the quoted one (`toContain("= 42")` plus `not.toContain('"42"')`), or match on the AST instead of the printed text.
- **Task 12's "keeps a filler's bindings local to the hole" asserts nothing about scoping.** It checks the output *contains* `const inner = 5`, which is true under plain splicing with no isolation at all — i.e. true precisely when the rule is broken. The rule's teeth would be: template code *after* the hole references `inner`, and the completed program must fail to resolve it (or hygiene must have renamed it). No such test exists anywhere in the plan.
- **Task 12's execution test never reaches the feature.** Covered in the blocking section: explicit `return` in the fixture, and the check defaults to `"warn"`. It passes before Step 4 is written.
- **Task 14's assertion can be satisfied by a destroyed value — almost.** `holesOf(filled).length` is `0` for any object whose walk finds nothing, including a `Code` value that came back as garbage. It's rescued *by accident*: `fill` throws first when `value` isn't among the (empty) holes. Make the rescue deliberate — assert on the filled program's printed content, not just the hole count.
- **Task 11's two-filler collision test doesn't require the fresh names to differ.** `match(/const (__hyg\w+|tmp)\b/g)?.length === 2` is satisfied when both fillers get renamed to the *same* `__hyg1_tmp` — which is still a collision. Add an assertion that the two binder names in the output are distinct.
- **Task 15 doesn't test what its filename promises.** `generatedProgramHandlers` claims "the parent's handler still governs what it can do," but the driver has no `handle` block and the generated program attempts no guarded action. The injection half is well tested; the handler-governance half is untested. Add a variant: template imports a tool, an identifier hole picks it, the parent registers a handler, and the test asserts the handler saw (or rejected) the child's call. Handlers are safety infrastructure here — this is the one test the feature's premise rests on.

### Coverage the plan admits is missing, and coverage it doesn't know is missing

- **The plan's own self-review note 4 concedes hygiene has untestable gaps and then doesn't close them.** "Parameters and for-loop binders need the same treatment, and no test in the plan will catch their absence" — the remedy is to *add those tests* (filler colliding with a template function parameter; filler colliding with a for-loop binder), not to read three type files and hope. As written, `bindersOf` regressing to assignments-only can never be caught.
- **No test fills a splice.** Task 6 tests splice *parsing* and *printing*; nothing ever calls `fill` on a `#...items` hole. `fillOne`'s splice branch — the array requirement, the flatMap, the spread into the surrounding statement list, and the "splice needs an array" error — is entirely unexercised. Since "one import per selected tool" is the plan's stated primary use case for splices, this is the feature's headline scenario with zero fill-time coverage.
- **String-escaping injection is untested beyond one embedded quote.** The security property is that a filler string stays inert through *print → re-parse in the subprocess*, which depends entirely on `generateAgency`'s string-literal escaping. Agency strings have interpolation; `liftValue` builds `segments: [{ type: "text", value }]`. If the generator prints a text segment containing the interpolation opener without escaping it, a filler like `"{ readFile(\"x\") }"` re-parses as an *interpolation* — code execution through the front door the lifting rule locked. Nothing tests fillers containing interpolation syntax, backslashes, newlines, or the delimiter. Add a battery: for each nasty string, fill → print → re-parse → the parsed literal's value equals the original string exactly. This is the same tier of importance as the Task 9 injection test.
- **Identifier holes accept reserved words.** `LEGAL_IDENTIFIER` matches `if`, `def`, `node`, `return`. `fill(t, { name: "if" })` produces `def if(): ...`, which explodes at re-parse in the subprocess with a confusing error far from the cause. No test covers it; the validator should check against the keyword list, and a test should pin the rejection.
- **Identifier holes bypass the reserved-prefix check.** Task 11's `assertNoReservedPrefix` runs on `Code` fillers only; `fill(t, { tool: "__hyg1_x" })` inserts the reserved prefix via a plain string. One test, one guard clause in `identifierNodeFor`.
- **No test parses a hole in binary-expression operand position.** `holeParser` is wired into "the top-level expression alternation," but the binop parser does precedence climbing with its own operand parsing (`docs/dev/binop-parser.md`). Whether `#a + 1` or `if (#cond && x)` parses at all depends on wiring the plan never mentions. *Unverified inference* — I have not traced the binop operand parser — but that is exactly why the test belongs in Task 2: if the wiring is fine the test is cheap, and if it isn't, no existing test notices.
- **Nothing tests the pipeline stages between the parser and the TypeScript builder.** Task 7 guards `typescriptBuilder.processNode`, but a hole first flows through `SymbolTable.build`, `buildCompilationUnit`, and the `TypescriptPreprocessor`. If any of those throws on the unknown node kind, the AG8001 test fails with the wrong error — and there's no test telling you which stage to fix. Add a unit test per stage: each either tolerates a hole or produces its own named diagnostic.
- **Negative parser cases:** `#` with no name, `#123`, `#""` (empty quoted name), `#...` with no name, and — post-Task 1 — a permanent test that `#` in a record type is a parse error (Step 8 verifies this manually with a scratch file, then deletes the evidence; nothing stops a regression).
- **Smaller gaps:** `node #n()` (only `def #name()` is tested, but Task 5 wires both); filling the *result* of a partial fill (fill composition — two sequential fills, then `holesOf` is empty); `loadTemplate` on a file that doesn't parse and on a missing file (error path of the `Result`); an AG8001 test with *two* holes verifying the `{names}` plural in the message (the builder throws on the first hole it reaches, so the message's plural can never be satisfied — either the diagnostic collects all holes before throwing, which needs a pre-pass, or the message should be singular); annotated-hole round-trip with a non-trivial type (`#items: string[] | null`).

### The AG8001/AG8002 execution tests can't be written at all

Already blocking (no `expectedError` in the runner), but it belongs in this list too because it means Tasks 7 and 13 currently have *no working test format* for their central deliverable. Whatever mechanism is chosen — runner extension or vitest against the compile path — the injection-adjacent diagnostics deserve first-class tests, not a format that silently can't express them.

## What's good and should be kept

Task ordering (free the sigil first, round-trip before more positions, injection test before everything downstream), the never-parse rule with its negative assertion as the first fill test, the two-round-trip formatter tests, committing migrations separately from the parser removal, and the self-review's honesty about sketch-level node shapes. Limitation 1 (error attribution) is correctly flagged as needing a decision before Task 13 — I'd add that the fragment-kind decision from the first blocking item should be made at the same time, since both change the `Code` representation.
